# BNI BINGO 紅黃綠燈平台 — Production Workflow

本文件分為兩部分：

1. LT 每月資料發布 SOP
2. 開發及版本更新流程

正式網址：<https://bni-traffic-light.bingo-win.workers.dev/>

---

## A. LT 每月資料發布 SOP

### 1. 準備 Excel

- 使用 BNI 系統匯出的原始 Member Traffic Light Excel。
- 檔名建議包含月份，例如：`Member Traffic Light - Bingo (Chinese) (2026-07).xlsx`。
- 不要先刪欄、改欄名、合併儲存格或自行重算分數。
- 確認檔案月份及會員名單正確。

### 2. 登入 LT 管理

1. 開啟正式網址。
2. 選擇「LT 管理」。
3. 使用 LT 專用密碼登入。
4. LT 密碼只供獲授權管理員使用，不應在會員群組公開。

### 3. 上載及自動核對

選擇 Excel 後，平台會：

- 自動選擇最符合 Traffic Report 結構的工作表
- 鎖定讀取 Excel 正式分數欄
- 保留原始 P、A、L、M、S、G、V、1-2-1、T、Biz Give 數據作改善建議
- 檢查會員姓名及 Excel 總分欄
- 檢查七項正式分數欄
- 檢查重複會員
- 檢查七項得分合計是否等於 Excel 總分
- 辨認及要求確認發布月份

任何必要欄位缺失、重複會員或七項合計不一致，發布按鈕都會被鎖住。

### 4. 發布前預覽

LT 必須核對：

- 檔案名稱
- 工作表名稱
- 發布月份
- 會員總數
- 每位會員七項分數、總分及燈號
- 系統判斷的發布模式

發布模式：

- **新月份**：成為最新月份
- **歷史月份**：加入紀錄，不改變最新月份
- **相同月份**：取代該月份全部會員資料

### 5. 正式發布

1. 確認月份。
2. 下載核對 CSV 作留底。
3. 按相應發布按鈕。
4. 如屬相同月份，確認取代提示。
5. 等待「發布成功」訊息。

平台只會將結構化資料保存到 Cloudflare D1；原始 Excel 不會上載或儲存於 Cloudflare，LT 必須另行保留原檔作核對及復原。

### 6. 發布後驗收

每次發布後必做：

- 最新月份卡片顯示正確月份及會員數
- 隨機抽查至少 3 位會員
- 抽查 1 位綠燈會員
- 抽查 1 位 1-2-1 未滿分會員，確認 1-2-1 可出現在改善方法
- 抽查 1 位 1-2-1 已滿分會員，確認列入「已達滿分」而非改善建議
- 確認「綠燈行動建議」全部行動的加分總和與預計總分一致
- 下載 1 張個人 PNG
- 下載一次「黃燈或以下」ZIP，核對人數及至少 2 張圖

完整清單見 [QA_CHECKLIST.md](QA_CHECKLIST.md)。

### 7. 派發改善圖

LT 後台「改善圖下載」預設為：

- 黃燈
- 紅燈
- 黑燈

綠燈會員預設不包括在改善圖內。LT 可按需要改為：

- 全部會員
- 只下載黃燈
- 紅燈及黑燈

下載按鈕會先顯示符合條件的會員數量。

### 8. 發現資料錯誤

如尚未發布：

- 取消上載
- 修正源頭 Excel 或重新匯出
- 再次上載及核對

如已發布：

- 不要直接改資料庫
- 使用相同月份重新上載
- 核對後使用「確認取代月份資料」
- 完成發布後重新做抽查及下載測試

## B. 開發及版本更新流程

### 1. 先寫驗收條件

任何修改前先清楚寫出：

- 誰會使用
- 要解決甚麼問題
- 哪些畫面／資料會受影響
- 成功標準
- 不可以破壞的既有功能

例：

> LT 可選擇只下載黃燈或以下會員圖卡；預設不包括綠燈；按鈕顯示實際人數；個人 PNG 不受影響。

### 2. 先分類改動風險

- **Data P0**：Excel 欄位、計分、月份、發布、Replace
- **Auth P0**：登入、session、權限
- **Core P0**：會員頁、PNG、ZIP
- **UX P1**：排版、文字、手機顯示
- **Enhancement P2**：新 dashboard、模擬器、分析功能

P0 改動必須完成完整 regression test。

### 3. 修改原則

- Excel 正式分數不可由前端重算覆蓋。
- 加分建議只可使用 raw metrics。
- 畫面列出的 Green Path 行動必須等於預計總分所計算的行動。
- 不要在多個檔案重複實作同一邏輯。
- 新外部 dependency 必須確認檔案已部署及有 fallback。
- 每次 production release 都要更新 cache-bust version。

### 4. 建議 Git 流程

未來重大修改建議：

1. 建立 issue／change brief
2. 建立 feature branch
3. 完成功能及文件
4. 部署 preview
5. 依 QA checklist 驗收
6. Merge 到 `main`，由 GitHub Actions 部署到 Cloudflare production
7. 核對正式網址版本
8. 更新 DECISION_LOG／LESSONS_LEARNED

小型文字或純 CSS 修正可直接提交，仍需核對 production。

### 5. 發布後檢查

- 正式根網址載入最新 cache-bust version
- API `/api/bni` 可回應 POST
- Cloudflare `/health` 回傳 `ok: true`，D1 可讀取
- 手機及 desktop 均無水平溢出
- 所有下載功能產生可開啟檔案
- 文件與實際功能一致

## C. 每月責任建議

- **資料負責人**：匯出及確認 Excel
- **LT 發布人**：上載、預覽、發布、下載核對 CSV
- **第二覆核人**：發布後抽查會員及 ZIP
- **技術維護人**：只在出現錯誤或功能更新時介入

同一個人可以兼任，仍建議發布後由另一位 LT 抽查一次。
